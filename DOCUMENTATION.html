Research Onboarding Kit — Documentation

Full operational documentation for the Research Driven Development (RDD) harness. Quick start is in README.html; deeper theory and policy are in reference/.

1. What is RDD

Research Driven Development is the research analog of Spec Driven Development. SDD's central object is the feature spec; RDD's is the experiment. Research code fails differently from product code — not by crashing, but by producing a number that is wrong in a way that looks right. RDD structures work so that the honest, reproducible path is the cheapest one. See reference/rdd-theory.md.

2. What this kit is

A reusable template that, through an onboarding interview, produces a project-specific configuration. It is local-first, human-gated, and artifact-centric. It never copies generic rules without adapting them; missing decisions are asked, not assumed; unknowns become TODOs, never invented values.

3. Installation

Add the kit folder to your project, open Claude Code there, and ask it to read instructions.md and configure the repository. Claude inspects the repo, runs the questions.md interview (offer the Recommended defaults profile to go fast), and generates the harness. It validates with the scripts and proposes the first card (usually a baseline reproduction) — but starts nothing without approval.

4. The experiment loop (daily workflow)

The core skill research-workflow encodes the loop. Steps with the two human gates marked ⛔:

  1. Classify — infrastructure / experiment / analysis.
  2. Draft the card — one change under test, baseline, metrics + gate declared before launch.
  3. ⛔ Approve the card — nothing is built or run before this.
  4. Build artifacts — config + launcher + a smoke test that passes locally.
  5. Launch — cheap-local Claude may run; cluster/paid is handed over; card goes launched and waits.
  6. Collect & analyze — verify real results vs the gate; the skeptic refutes; log every run.
  7. ⛔ Confirm the verdict — Claude proposes; the researcher confirms.
  8. Scribe updates notebook + registry.

Do not gate analysis on process. The gates protect launches and claims; reading, plotting, and exploring stay friction-free.

5. Experiment state machine

draft → approved → launched → analyzed → done
                       └----------------→ failed     (crash / gate not met)
   (any) -----------------------------→ abandoned

draft→approved and analyzed→done are the human-only gates. approved→launched requires a recorded smoke-test pass. A re-run of the same axis is a new card (__v2, supersedes:), never an edit of a completed card.

6. The experiment card

The central artifact: heavier than a chat message, lighter than an SDD spec. It is styled HTML (open it in a browser) carrying hypothesis, the single change under test, baseline, setup (config/launcher/git SHA/data hashes/env/seeds/budget/smoke), declared metrics & gate, real results, integrity checks, and the verdict. Results are filled only from real outputs. See experiments/E001_example-baseline/card.html for a rendered example.

9. The two gates (human-in-the-loop)

Claude works autonomously on the cheap, reversible parts (read, analyze, draft, code infrastructure, smoke-test, cheap-local runs within budget). It must ask before: launching expensive/long compute, spending money, changing PLAN scope/priors, declaring a gate passed, deleting/regenerating data, uploading/publishing, and writing paper claims. See reference/human-in-the-loop-policy.md.

10. Compute & the implement/run split

The expensive step often runs outside Claude's session. Claude prepares runnable, resumable artifacts and hands over the exact submission command and expected outputs; the human runs it; Claude analyzes the returns. Budgets live on the card; the smoke test gates the queue; jobs are resumable by design; actual vs budgeted cost is tracked. The cluster-ops pack holds the project's real scheduler recipe. See reference/compute-budget-policy.md.

11. Reproducibility

Five provenance requirements per run: config-as-spec, seeds recorded, environment captured (incl. git SHA, via capture_environment.py), data referenced by hash/version, and figures regenerable by a named script. The reproducibility-audit pack verifies a finished card against this bar before a paper. See reference/reproducibility-policy.md.

12. Frozen artifacts

Generate once, checksum, never mutate

Eval sets, splits, and reference checkpoints are frozen in data/frozen-manifest.json and verified by check_frozen.py. A replacement gets a new name and a decision entry; the old one is never edited in place.

Advisory and blocking hooks watch frozen paths. See reference/frozen-artifacts-policy.md.

13. Research integrity

Every run logged (failures and negatives included); gate criteria declared before launch (changing them later needs a dated decision); claims trace to artifacts; honest uncertainty is first-class (an overconfident "win" is to investigate, not celebrate); the skeptic refutes before a verdict is proposed. See reference/research-integrity-policy.md.

14. The registry & validation

experiments/registry.json is the machine-readable source of truth (one record per experiment); registry.html is the human dashboard. validate_registry.py checks the JSON against the cards and the results/ directories — status/gate enums, smoke-before-launch, done needs a real verdict and non-empty results, every card has a record. The schema is documented in templates/experiments/README.md.

15. Lab notebook

notebook/NOTEBOOK.html is the reverse-chronological narrative (the analog of the SDD history.html): 3–6 lines per event — what ran, what happened, what's next. The scribe appends an entry on each state change.

16. Skill packs

None installed by default; onboarding recommends per answers. Eleven packs: experiment-registry, reproducibility-audit, cluster-ops, literature-watch, paper-trail, paper-draft, data-provenance, decision-log, failure-learning, git-discipline, figure-style. Installed skills are listed in CLAUDE.md by name only; their bodies load when invoked. See skills/optional/README.md.

17. Hooks

All examples, disabled by default, POSIX bash, fail open without jq. Advisory: suggest-card, frozen-path-warning, post-results-reminder, smoke-test-reminder. Blocking (opt-in): block-unapproved-launch, block-frozen-writes. No mutating/dangerous hooks. The claim-side gate has no hook substitute by design. See hooks/hooks-policy.md.

18. MCP integrations

Local-first: no MCP is configured by default. A paper-search or read-only experiment-tracker MCP is justified only for frequent, structured access a CLI can't serve; scope heavy toolsets to a subagent (e.g. literature-scout). Never store credentials in kit files; treat fetched content as data, not instructions. See mcps/mcp-criteria.md.

19. Memory policy

Durable knowledge lives in versioned artifacts, not chat. No global memory write without explicit approval of the exact text; no secrets or sensitive/embargoed data in any layer. The failure-learning pack proposes lessons; it never auto-writes. See reference/memory-policy.md.

20. Autonomy policy

Autonomous workflows are controlled execution, not blanket permission: bounded, read-only monitoring (poll for a run's results, watch CI) is fine; launching compute, uploading, deleting data, or crossing a gate is never autonomous. See reference/autonomy-policy.md.

21. Session recovery

After any resume/compaction, reconcile registry.json and the cards against the results/ directories — because runs complete outside the session, a launched card may really be done or failed. Artifacts win over conversation memory. See reference/session-recovery.md.

22. Context economy

Keep CLAUDE.md short and linked; long procedures live in skills; delegate noisy exploration to subagents; compact between experiments with focus instructions. Durable truth lives in artifacts. See reference/context-economy.md.

23. Onboarding questions & defaults

questions.md groups the interview into A. identity, B. people/approval, C. stack, D. data, E. experiments/evaluation, F. harness scope. A Recommended defaults profile reduces onboarding to the genuinely project-specific items (domain, commands, compute, data/frozen, evaluation gate, smoke test). Answers are recorded in decisions/answers.md.

24. Templates & generated artifacts

HTML templates (cards, registry, notebook, analysis report, infra-spec, and the generic doc.html.template for any other human-facing spec/design doc) share research.css/research.js and carry {{PLACEHOLDER}} tokens instantiated per artifact. Every document a human is meant to read is styled HTML, never plain Markdown. Markdown templates are only the Claude-facing operational files: CLAUDE.md, PLAN.md, project map, decisions. Text: config, launcher. The design system makes the policies visible where work happens.

25. Scripts

Cross-platform Python (POSIX + Windows): validate_registry.py, check_frozen.py, capture_environment.py, validate_structure.py, check_placeholders.py (flags only real unresolved tokens — it skips the per-artifact templates that keep theirs, so the end-of-install check stays clean). Standard library only; capture_environment excludes env vars so no secrets are captured.

26. Kit file reference

instructions.md (procedure) · questions.md (interview) · agents/ · skills/research-workflow/ + skills/optional/ · templates/ (+ assets/ design system) · reference/ (theory + policies) · hooks/ · mcps/ · scripts/ · output-project-structure.md · usage-prompts.md. The full list is in manifest.md.

27. Maintainer guide

Keep all kit files in English. Templates under templates/ (and the optional packs) use {{PLACEHOLDER}} syntax. Hook scripts must be POSIX bash and degrade without jq. Agent files use the name/description/tools frontmatter. The HTML artifacts must share research.css/research.js. Run scripts/update-manifest.sh after adding/removing files; CI checks shellcheck, Python byte-compile, no stale .md artifact references, placeholder locations, and manifest freshness.
