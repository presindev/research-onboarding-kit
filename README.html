Research Onboarding Kit for Claude Code

Installs a Research Driven Development (RDD) harness in any repository — the research-focused sibling of the sdd-onboarding-kit. For full operational docs, open DOCUMENTATION.html.

What it is

A reusable template that, through an onboarding interview, produces a project-specific configuration: CLAUDE.md, PLAN.md, .claude/agents/, .claude/skills/, experiments/ (cards + registry), notebook/NOTEBOOK.html, decisions/, and validation scripts. It is not a global configuration.

English is canonical for kit files; generated artifacts may use the project's language.

The experiment loop

RDD replaces the SDD feature-spec loop with the experiment loop, built on the experiment card, with two human gates (⛔):

  1. classify + draft
    Classify (infra/experiment/analysis); draft the card — one change under test, baseline, metrics + gate declared before launch.
  2. ⛔ approve
    Researcher approves the card. No config/launcher/run before this.
  3. build + launch
    Config + launcher + smoke test (passes locally first). Cheap-local runs Claude may do; cluster runs are handed over and the card waits — no fabricated results.
  4. analyze
    Verify real results vs the gate; the skeptic tries to refute; every run logged, failures and negatives included.
  5. ⛔ confirm
    Researcher confirms the verdict; scribe updates notebook + registry.

The two gates sit where research goes expensively or dishonestly wrong: launching compute and making claims. Analysis stays friction-free.

Why these mechanisms

Distilled from a real case: ~6 months lost to simultaneous untested changes, a silently corrupted validation set, and checkpoint selection keyed to it.

Failure modeCountermeasure
Many simultaneous changes; unattributableOne change per experiment
Eval data silently mutated / leaksFrozen-artifact policy + check_frozen.py
No baselineBaseline & reproduction gate first
Cherry-picking (p-hacking)Every run logged, negatives included
Not reproducible laterConfig-as-spec, seeds, env capture, data hashes
Compute wasted on bugsSmoke test before the queue
Claims not traceableClaim → card → artifact at write-up

How to install

Add this folder to your project, open Claude Code there, and run:

Read `research-onboarding-kit/instructions.md` and configure this repository to
use Research Driven Development. Ask me all necessary questions before making
project-specific decisions.

What gets installed

By default: a concise CLAUDE.md + PLAN.md; six agents; the research-workflow skill; experiments/ (registry.json + registry.html + the shared design system); notebook/NOTEBOOK.html; decisions/answers.md; the project map; and the cross-platform validation scripts.

Optional (opt-in): eleven skill packs (experiment-registry, reproducibility-audit, cluster-ops, literature-watch, paper-trail, paper-draft, data-provenance, decision-log, failure-learning, git-discipline, figure-style); example hooks (disabled by default); paper-search/tracker MCPs. The default install is fully local.

Artifacts: styled HTML + machine state

Durable human-facing artifacts (cards, registry dashboard, notebook, analysis reports) are styled, self-contained HTML sharing one design system. Experiment state lives in experiments/registry.json — the validation source of truth, kept separate from the HTML. Operational files (CLAUDE.md, skills, agents, policies) are Markdown; configs and launchers are text.

See the rendered reference example at experiments/E001_example-baseline/card.html.

Safety model

  • Two human gates Claude never crosses on its own (card approval, verdict confirmation).
  • No expensive/long compute without approval; smoke test before any cluster job.
  • Frozen artifacts never mutate; replacements get new names + a decision.
  • Every run logged; no metric shopping; claims trace to artifacts.
  • No global memory write without explicit approval; no secrets/sensitive data in any layer.
  • Hooks advisory/blocking, never mutating/dangerous, disabled by default; autonomy is read-only monitoring at most.

Contributing

Contributions are welcome. To propose a change:

  1. Fork the repository.
  2. Create a branch from main with a descriptive name (e.g. fix/hook-path-filter or feat/cluster-ops-htcondor).
  3. Make your changes. Keep PRs focused — one concern per PR.
  4. Open a pull request against main with a clear description of what it does and why.
  5. Wait for review. Only the repository owner merges PRs.

Guidelines

  • Keep all kit files in English (generated project artifacts may use the project's language).
  • Templates under templates/ and the optional packs use {{PLACEHOLDER}} syntax.
  • HTML artifacts must share research.css / research.js — do not fork the design system.
  • Hook scripts: POSIX bash, fail open without jq.
  • Validation scripts: cross-platform standard-library Python (POSIX + Windows).
  • Agent files use the name/description/tools frontmatter; keep skeptic and literature-scout read-only.
  • Run scripts/update-manifest.sh after adding/removing files; commit the updated manifest.md.
  • CI must pass (shellcheck, Python byte-compile, stale-reference / placeholder / manifest consistency).

Reporting issues

Open a GitHub issue with a short description, steps to reproduce (if a bug), and expected vs. actual behavior.
